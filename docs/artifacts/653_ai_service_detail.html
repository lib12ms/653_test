04프롬프트에 내장된 "내부 3단계" 인지 지시

한 번의 API 호출 안에서, 모델에게 스스로 3단계를 밟도록 지시합니다 — 완전히 분리된 다단계 호출은 아니지만, 같은 문제의식(주제 파악과 제약 준수를 한번에 시키면 후자를 잊는다)에 대한 가벼운 대응입니다.

1단계핵심 주제 영역 파악

이 책의 핵심 주제 영역 2~3개를 먼저 식별

2단계내용어 → 주제어 치환

각 영역에서 검색 키워드를 뽑되, "책에 등장하는 사물·사례(내용어)"인지 "책이 다루는 개념(주제어)"인지 판별해 내용어는 주제어로 치환

3단계필터 규칙 적용

분야별 지침의 금지 규칙을 적용해 최종 목록 확정 — 이 단계의 실질적 집행은 모델이 아니라 05번의 후처리 코드가 다시 한번 담당

왜 "가벼운" 대응인가

진짜 다단계 아키텍처라면 후보 생성과 필터링을 서로 다른 API 호출로 분리해, 1차 호출은 제약 없이 풍부하게 뽑고 2차 호출이 사서 관점에서 엄격히 골라내는 방식일 것입니다. 지금은 한 호출 안에서 "단계를 밟으라"고 지시만 할 뿐이라 모델이 여전히 제약을 놓칠 수 있고, 그 부담을 05번의 정규식·차단어 후처리가 대신 떠안고 있습니다.

063단 계단식 폴백과 품질점수

finalize_653() — AI 결과가 부족할 때만 한 단계씩 아래로 내려가며, 내려갈수록 감점됩니다.

기본필터링항상 실행

AI가 낸 원시 키워드를 05번 카탈로그 규칙 + 제목/저자 금지어로 검증. 살아남은 것이 valid_keywords

2단계텍스트 fallbackvalid_keywords == 0 (문학 그룹 제외)

_extract_backup_candidates() — 설명·목차를 토크나이즈해 조사/어미 패턴을 걸러내고 보충. backup_used 플래그, -0.15점

3단계카테고리 fallback여전히 min_keywords 미달

_extract_category_candidates() — 분류 체인의 하위분야명, 그마저 없으면 _GENRE_FALLBACKS(장르별 대체어). category_fallback_used 플래그, -0.10점 (실제로 키워드가 추가된 경우만)

품질점수 공식

score = min(final_count, max_keywords) / max_keywords
if filter_rate > 0.3: score -= (filter_rate - 0.3) × 0.5
if backup_used: score -= 0.15
if category_fallback_used: score -= 0.10
score = clamp(score, 0.0, 1.0)

filter_rate = 차단된 키워드 수 / AI 원시 생성 수. 배치 스크립트는 이 점수가 0.55 미만이거나 최종 키워드가 4개 미만이면 "검토필요"로 분리합니다.
